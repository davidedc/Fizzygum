# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What Fizzygum is

A CoffeeScript GUI framework — a "web operating system" (windows, desktop, drag-and-drop, live in-system editing tools) rendered on a single HTML5 `<canvas>`. It descends from Morphic.js. ~470 `.coffee` files live in `src/`.

## Build, run, watch

Prerequisites are installed **globally, not via `npm`**: `coffee` (`npm i -g coffeescript`), `terser`, `python3`, and `fswatch` (watch script only). `npm test` is a do-nothing stub — see Testing.

- **Build:** `./build_it_please.sh` → output goes to `../Fizzygum-builds/latest`.
  - `--keepTestsDirectoryAsIs` — skip recopying tests; much faster, use while iterating.
  - `--homepage` — production build: strips tests + experimental code, ships a pre-compiled image.
  - `--notests` — drop tests, keep experimental code.
  - `--includeVideoPlayer --includeVideos` — bundle the video player + assets.
  - `--noSyntaxCheck` — skip the build-time CoffeeScript syntax gate (see Testing).
- **✅ Test a change — PREFERRED / DEFAULT:** `./build_and_test.sh` — full build **+** the whole SystemTest suite run **headless, in parallel shards, at `speed=fastest`, dpr 1** (~1 min on a many-core box). This is the standard way to verify a behaviour change; don't watch the suite in a browser (~15 min) or run tests one-by-one unless debugging a single test. One-time setup: `cd ../Fizzygum-tests && npm i` (Puppeteer).
- **Build + boot-smoke gate:** `./build_and_smoke.sh` builds (incl. the syntax gate) then headless-boots the world (native + SWCanvas), failing on any console error — a lighter boot-only check (use when a full suite run is overkill).
- **Watch + rebuild on save:** `./build_as_soon_as_anything_changes.sh` (`fswatch` over `src/`; the syntax gate runs on every save).
- **Run:** open `../Fizzygum-builds/latest/index.html` in a browser. It loads over `file://`; no dev server needed.
- **Find duplicated code:** `./find_duplicated_code.sh` — jscpd (devDependency) over `src/**/*.coffee`, no compile step (CoffeeScript is tokenized natively); console/JSON/markdown/HTML reports plus an LLM-handoff clone-pair list land in `duplication-report/` (gitignored). Tuning + gotchas (jscpd silently skips >1000-line files at its defaults — config raises the caps): `docs/tooling/duplicated-code-detection.md`.
- **Find SIMILAR (renamed/structural) code:** `./find_similar_code.sh` — jsinspect AST matching over a CoffeeScript-1-compiled ES5 mirror (`buildSystem/coffee-to-js-mirror.js`); catches the renamed/near-miss clones jscpd's exact-token scan can't. ⚠ its reports carry compiled-JS line numbers — navigate by file + `@method` name. Both scanners take `--tests` to scan the sibling `Fizzygum-tests` repo (harness + node scripts) instead. The recurring **re-audit → triage-ledger → LLM-session cycle** (incl. the `duplication-report/triage-report.md` ledger convention and its end-of-arc snapshot into `docs/done/`) is documented in the same docs file: `docs/tooling/duplicated-code-detection.md`.

The build **requires** this sibling layout and aborts without it:
```
Fizzygum-all/
  Fizzygum/          ← this repo (source only)
  Fizzygum-builds/   ← build output (generated; never hand-edit)
  Fizzygum-tests/    ← SystemTests + Automator source (separate repo)
  Fizzygum-website/
```

## Architecture (the non-obvious parts)

- **Subsystem references live in `docs/architecture/`** — present-tense, verified-against-src: the layout/settle engine ([`docs/architecture/layout.md`](docs/architecture/layout.md): the world cycle, settle tiers, ordered down-walk, constraint-box sizing, claimsSpace, the standing zero-invariants, the new-layout rulebook) and the affine transform islands ([`docs/architecture/transforms.md`](docs/architecture/transforms.md): TransformFrameWdgt islands, TransformSpec, the two-vocabulary law, pinned anchors, plane-mapped input, the island buffer cache). Read those before touching either subsystem.
- **Source-as-text + in-browser compilation.** `buildSystem/build.py` does *not* compile `src/` classes. It wraps each class's CoffeeScript source into an escaped JS string (`window.Foo_coffeSource = "…"`), grouped into ~12 `sources_batch_*.js` files; the browser compiles them at boot with the bundled CoffeeScript compiler. `--homepage` instead ships a `pre-compiled.js` image (generated by loading the page with `?generatePreCompiled`) for instant startup. Only `src/boot/*` is compiled to JS at build time.
- **No imports / no module system.** Zero `require`/`import`; every class is a global. Load order is auto-discovered by `src/boot/dependencies-finding.coffee`, which *regex-scans source text* for `extends X`, `@augmentWith X`, and `new X`. To reference another class just name it — but use those literal forms so the dependency finder sees the edge.
- **Class hierarchy:** `TreeNode` → `Widget` → `PanelWdgt` → … → `WorldWdgt`, the global singleton `window.world`. Widgets form a tree, painted recursively; a Widget with no owner is never drawn.
- **Meta classes.** `src/meta/Class.coffee` and `Mixin.coffee` parse each source at runtime, which is what enables live inspection/editing of any class from inside the running world.
- **Rendering** is a *broken-rectangles* (dirty-region) repaint loop on the canvas. A widget invalidates only ITSELF — `@_changed()` (just this widget) / `@_fullChanged()` (it + its subtree), both PRIVATE: never call them on another widget (widget-citizenship point 2, gated by `check-invalidation-receivers.js`); cross-object repaint goes through intent-named public methods on the receiver (e.g. `world.noteWallpaperChanged()`, `world.caret.noteTextChanged()`) — there is no general-purpose public repaint verb. Pluggable `*Appearance` objects do the drawing; widgets opting into `BackBufferMixin` cache themselves to an offscreen canvas.
- **Software 3D (SW3D) for Fizzytiles.** The `src/fizzytiles/` app ("fridge magnets" / Fizzytiles) runs the LiveCodeLang program its tiles compile to and **software-renders** it — no WebGL, no twgl (deleted). It uses **SW3D**, the userland software-3D engine (`examples/sw3d.js`) vendored from the SWCanvas repo alongside `swcanvas.min.js` (see `scripts/vendor-swcanvas.sh` + the boot-bundle assembly in `build_it_please.sh`; both are bundled in *every* build, gated by the `vendor/swcanvas.pin` SHA). `FridgeMagnets3DCanvasWdgt` rasterizes each frame into an `SWCanvas.Core.Surface` (+ `DepthBuffer`/`Triangle3DOps`) and blits it; its matrix stack is `LCLTransforms`. Determinism: the animation clock is event time under the Automator (`docs/archive/fizzytiles-sw3d-port-plan.md`).
- **Dataflow / spreadsheet engine — the SECOND drain station.** `WorldWdgt.doOneCycle` runs **two parallel drain stations**: `recalculateDataflow` (settles VALUES) runs BETWEEN `runChildrensStepFunction` and `recalculateLayouts` (settles GEOMETRY), and the coupling is one-way — dataflow may dirty layout, never the reverse. ONE engine (`world.dataflow`, `src/dataflow/`) serves two clients: **spreadsheet** cells wired by formula references and **patch-programming** widgets wired by connections (`@target`/`@action`). A change marks nodes **stale**; values are **pulled** at recompute time (notifications carry no values); the forward/reverse edge index is derived and never serialized. Design: [`docs/specs/dataflow-engine-spec.md`](docs/specs/dataflow-engine-spec.md); depth: [`src/dataflow/CLAUDE.md`](src/dataflow/CLAUDE.md) + [`src/spreadsheet/CLAUDE.md`](src/spreadsheet/CLAUDE.md); measured convergence (typical 1 pass): [`docs/measurements/dataflow-measurements.md`](docs/measurements/dataflow-measurements.md).
- **Window drag-embed (dwell-to-arm).** Float-dragging a *window* over a receptive container embeds it only after a ~450ms **dwell** (held still within 7px; `ActivePointerWdgt` state machine — `dwellToArmMs`), so a window never nests by accident; an unarmed release lands it on the desktop, and dragging a nested window out ejects it. Plain (non-window) payloads still embed instantly (the payload-class rule, `requiresDeliberateEmbedding`). A window's internal-vs-external skin is **DERIVED from its parentage** (`FrameWdgt.isInternal` = nested in a real container), re-derived on every re-parenting — there is no stored flag and no manual switch button. All drag feedback is ephemeral overlays. Design + as-built deviations: [`docs/specs/drag-embed-interaction-spec.md`](docs/specs/drag-embed-interaction-spec.md); arc history: [`docs/archive/drag-embed-implementation-plan.md`](docs/archive/drag-embed-implementation-plan.md).
- **Mixins** are applied with `@augmentWith SomeMixin` (see `src/mixins/`). The whole widget tree can be duplicated (and, via the `src/serialization/` `Serializer`/`Deserializer` pair, serialized to a versioned JSON file) through the one graph copier `DeepCopierMixin` — that is also how SystemTests snapshot state. **All serialization / deserialization / duplication internals — format, traversal contract, per-class protocol, `file://` I/O — are documented in one place: [`docs/architecture/serialization-duplication-reference.md`](docs/architecture/serialization-duplication-reference.md).** Mixins are being **phased out in favour of plain-OO delegation**; the first example is `MacroToolkit` (`world.macroToolkit`, in `src/macros/`), the macro-test helper toolkit split out of `WorldWdgt`.

## Testing

Two fast automated checks complement the (manual/browser) SystemTests:
- **Build-time syntax gate** (`buildSystem/check-coffee-syntax.js`, run automatically by `build_it_please.sh`): since the ~470 class/mixin sources ship as text and compile in-browser, this is what catches CoffeeScript *parse* errors at build time. It LOADS the real `src/meta/Class.coffee`/`Mixin.coffee` and compiles each source the same *fragmented* way the browser does — a whole-file `CoffeeScript.compile(src,{bare:true})` is NOT faithful (it false-fails on most files). Skip with `--noSyntaxCheck`.
- **Headless boot smoke** (`../Fizzygum-tests/scripts/smoke-boot-headless.js`, via `./build_and_smoke.sh` or `cd ../Fizzygum-tests && npm run smoke`): boots the built world headless (native + SWCanvas) and fails on any console/page error — catching the load-order/runtime faults the syntax gate cannot.

Behavioural tests are **SystemTests**: each one drives the running world from a high-level **macro** (a generator that asks the live world where things are and synthesises real input events), then compares canvas screenshots pixel-by-pixel against reference images. The Automator source and 265 tests live in the sibling `Fizzygum-tests` repo (not here); any non-`--homepage` build copies them in. (Macros are the only authoring path — the old input *recorder* has been removed.)
- **Author:** write the test directly as a macro — the framework-side helper toolkit lives in **`src/macros/`** (engine `Macro.coffee` + `MacroToolkit.coffee`, reached as `world.macroToolkit`) and is documented in **`src/macros/CLAUDE.md`**; see also the `/author-macro-test` skill in `Fizzygum-tests`.
- **✅ Run the whole suite — PREFERRED / DEFAULT:** `./build_and_test.sh` (build + suite), or against an existing build `cd ../Fizzygum-tests && npm test` (= `scripts/run-all-headless.js`). Runs all 265 tests **headless, in parallel shards, at `speed=fastest`, dpr 1** — ~1 min on a many-core box (vs ~15 min watching a browser). `npm run test:all` is the single-process variant. This is the default way to verify a change. The suite can also run **headless under Safari/WebKit** (Chrome stays default) as a cross-engine determinism check — `--browser=webkit` / `npm run test:webkit`, reusing the same references; see `../Fizzygum-tests/CLAUDE.md`.
- **Run one (debugging a single test):** open the built `worldWithSystemTestHarness.html`, then `world.automator.loader.loadAndRunSingleTestFromName('SystemTest_name')`; or headless `node scripts/run-macro-test-headless.js SystemTest_<name>`.
- **Determinism (the suite asserts byte-exact pixels — keep it deterministic):** render/layout/input code must be a pure function of the **event stream** and **final geometry**, never of wall-clock timers (`setTimeout`/`Date.now`), frame/cycle counts, or intermediate layout passes — those diverge at **dpr 2 under parallel load** (heavy cycles starve timers and drain multiple events per frame). The contract, the recurring bug-class with worked case law, and the diagnosis playbook (deterministic heavy-cycle reproduction, pixel-delta forensics, disable-the-mechanism proof) are in **`../Fizzygum-tests/DETERMINISM.md`** — read it before touching the rendering loop, `_reLayout`, or `ActivePointerWdgt` input recognition.

## Conventions & gotchas

- **`docs/` is bucketed by what a file IS — keep the split honest** (map + full filing rules: `docs/README.md`): `architecture/` how it works NOW · `specs/` behaviour contracts · `plans/` ACTIVE arcs only · `archive/` closed/falsified/parked plans, verbatim + status-stamped, indexed in `archive/INDEX.md` · `measurements/` dated snapshots · `tooling/` dev-tool how-tos. The standing loop: new arc → `plans/`; arc closes → `git mv` to `archive/` + stamp + INDEX line; durable residue → the subsystem's `architecture/` doc **in the same arc**; every open/deferred item gets a `docs/BACKLOG.md` line pointing at its owning plan section.
- **One class per file; filename must equal the class name** (`Widget.coffee` ↔ `class Widget`). `build.py` keys off this.
- **`Wdgt` is the modern name for the legacy `Morph`** (both extend `Widget`). **The intended end state is all-`Wdgt`** — `Morph` is legacy. New widgets are `*Wdgt` from the start, and existing `*Morph` classes ARE meant to migrate to `*Wdgt` over time. Do it **incrementally, not in one sweep**: rename a coherent group (a class + its family) when you're already touching it, with a reason, because (a) a rename here is *not* always pixel-free — menu/hierarchy labels strip `Wdgt`, so renaming a class whose colloquial name is drawn (e.g. menu items) shifts that label and forces SystemTest screenshot recapture; verify per class — and (b) each rename is a whole-tree identifier+file+serialization sweep, best done one verifiable batch at a time.
- **`nil` means `undefined`** (defined in `src/boot/globalFunctions.coffee`); the codebase uses it instead of `null`/`undefined`.
- **Every widget is PLACED and SIZED in integer pixels — enforced.** Bounds round at the *desired* funnel (`Widget._moveToNoSettle`/`__commitExtent`) AND at every layout *producer* that computes a fractional target (axis ticks, horizontal-stack distribution, `center()`-based centring, corner-internal sizing, …); the always-on guard `Widget._assertBoundsWellFormed` (called from each bounds-commit leaf) `console.error`s `NON_INTEGER_GEOMETRY` (fractional applied `@bounds`) and `NON_FINITE_GEOMETRY` (NaN/Infinity), **both wired into the headless runners' fail-gate** — a fractional placement fails the suite even when the pixels match. Fractional *desired* geometry is kept on the side (e.g. `positionFractionalInHoldingPanel` for stretchable panels); internal *content* rendering (vector icons, charts, rotated strokes) is legitimately fractional. Integer placement is **necessary but not sufficient** for a back buffer to be byte-identical to a direct draw. Policy + rationale + the back-buffer FP caveat: [`docs/architecture/integer-pixel-placement-and-sizing.md`](docs/architecture/integer-pixel-placement-and-sizing.md); the enforcement arc + the divider-drag reproportion sub-pixel-sensitivity case-law: [`docs/archive/fractional-widget-bounds-investigation-plan.md`](docs/archive/fractional-widget-bounds-investigation-plan.md). Widgets are placed and sized in integer pixels **in their own plane**; under an affine `TransformFrameWdgt` island the *screen-family* queries (`screenBounds()`/`localPointToScreen()`/…, every name contains `screen`) are DERIVED, post-transform, and possibly fractional, while the layout-box family (`width`/`height`/`bounds`/`center`/…) stays plane-local and integer — the two-vocabulary law, see [`docs/archive/affine-geometry-api-plan.md`](docs/archive/affine-geometry-api-plan.md).
- Edit only `src/**/*.coffee`. Never edit `../Fizzygum-builds/**` — it is regenerated every build.
- **Never `git stash` in this repo.** On 2026-07-05 a `stash pop` silently emptied both the working tree AND the stash list (a `.gitattributes` warning is implicated); recovery required digging git dangling blobs out by hand. Use a WIP commit or a `git worktree` for clean-tree A/B comparisons.
- `# … excluded from the fizzygum homepage build` comments and `if Automator?` guards mark test/experimental code that `--homepage` strips as dead code — keep them intact.
- For subsystem-specific depth, check for a local `CLAUDE.md` in that subdirectory (e.g. `src/boot/`, `src/meta/`, `src/patch-programming/`) before assuming this file is complete.
